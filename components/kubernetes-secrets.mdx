---
title: "Kubernetes secret provider for per-target scrape auth"
description: "The google/secrets package watches Kubernetes Secrets and supplies per-target credentials at scrape time, keeping sensitive auth values out of prometheus.yml."
---

In large multi-tenant clusters you may need to scrape hundreds of targets that each require different credentials — for example, per-tenant bearer tokens or basic auth passwords. Storing all of those credentials directly in `prometheus.yml` is insecure and impractical to rotate. The Kubernetes secret provider in the `google/secrets` package solves this by watching Kubernetes Secrets directly from the Prometheus process and making their values available as named references in scrape configuration.

## How it works

The provider maintains a **watch loop** per referenced Kubernetes Secret using the Kubernetes API `Watch` mechanism. When Prometheus starts (or hot-reloads its configuration), the `Manager` calls `ApplyConfig`, which:

1. Creates a Kubernetes client using in-cluster config (or a provided kubeconfig / API server URL).
2. For each `SecretConfig` entry, starts a `secretWatcher` goroutine that calls `client.CoreV1().Secrets(namespace).Watch(...)` filtered to the specific secret name.
3. Caches the latest `corev1.Secret` object in memory. On `ADDED` or `MODIFIED` events the cache is updated; on `DELETED` the cache is cleared.
4. If the watch connection drops (network issue), the watcher automatically restarts with a random jitter between 1–31 seconds.

When Prometheus needs a credential value during a scrape (via `Manager.Fetch`), the provider reads the cached secret value in memory — there is no extra API call at scrape time.

The provider exposes two Prometheus metrics for operational visibility:

- `prometheus_kubernetes_failed_secret_configs` — number of secret configurations that failed to load
- `prometheus_kubernetes_secrets_configured` — total number of configured secrets

<Note>
  The `google/secrets` package will be deprecated once [PROM-47](https://github.com/prometheus/proposals/pull/47) is implemented upstream. At that point the upstream Prometheus secret manager will provide equivalent functionality.
</Note>

## Configuring secret references

Secret references are defined in two places: a `secrets` section in `prometheus.yml` that maps logical names to Kubernetes Secrets, and references to those logical names inside `scrape_configs`.

### Secret configuration struct

Each entry in the `secrets` list has the following fields (from `google/secrets/manager.go` and `google/secrets/kubernetes.go`):

```yaml
# Top-level in prometheus.yml
google_cloud:
  secrets:
    kubernetes:
      # Optional: URL of the Kubernetes API server. Defaults to in-cluster config.
      api_server: ""
      # Optional: Path to a kubeconfig file. Defaults to in-cluster config.
      kubeconfig_file: ""
    secret_configs:
      - name: my-target-token       # Logical name used in scrape_configs
        config:
          namespace: monitoring     # Kubernetes namespace of the Secret
          name: target-credentials  # Name of the Kubernetes Secret object
          key: token                # Key within the Secret's data map
```

### Kubernetes Secret YAML

Create the Kubernetes Secret that holds the credential value:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: target-credentials
  namespace: monitoring
type: Opaque
data:
  # Base64-encoded value of the bearer token
  token: <base64-encoded-token>
```

### Referencing a secret in prometheus.yml

Use the logical name (the `name` field from `secret_configs`) in place of a literal credential value in your scrape configuration:

```yaml
scrape_configs:
  - job_name: my-secure-target
    authorization:
      type: Bearer
      credentials: $my-target-token
    static_configs:
      - targets:
          - secure-target.example.com:9100
```

<Info>
  The `$name` syntax resolves the named secret at scrape time using the cached in-memory value. The credential is never written to disk by Prometheus.
</Info>

## RBAC requirements

The Prometheus ServiceAccount must have permission to `get`, `watch`, and `list` Secrets in each namespace where referenced Secrets live. Apply a Role and RoleBinding like the following in each monitored namespace:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: prometheus-secret-reader
  namespace: monitoring
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "watch", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: prometheus-secret-reader
  namespace: monitoring
subjects:
  - kind: ServiceAccount
    name: prometheus
    namespace: prometheus-system
roleRef:
  kind: Role
  name: prometheus-secret-reader
  apiGroup: rbac.authorization.k8s.io
```

<Warning>
  Granting access to Secrets in a namespace gives Prometheus read access to **all** Secrets in that namespace. Use dedicated namespaces for per-tenant credentials and restrict the Role's `resourceNames` field to the specific Secret names where possible.
</Warning>

## Limitations

- The provider requires the Prometheus process to have network access to the Kubernetes API server.
- Duplicate `name` values within `secret_configs` are rejected — both entries are disabled and an error is returned.
- If the referenced Kubernetes Secret does not exist or is forbidden, the watcher retries automatically; scrapes that require the missing secret will fail with an error until the secret becomes available.
- **This feature will be deprecated** once [PROM-47](https://github.com/prometheus/proposals/pull/47) is implemented in upstream Prometheus. Plan to migrate to the upstream secret manager when that proposal lands.

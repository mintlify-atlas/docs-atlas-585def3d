The GMP Prometheus fork uses the same `prometheus.yml` configuration file as upstream Prometheus, with one important addition: you must set GMP-specific labels in the `global.external_labels` block. These labels identify the Google Cloud project and region to which metrics are exported via the in-process GCM gRPC pipeline.

## Required external labels

GMP requires two external labels to be set before any scrape or rule evaluation begins. Without them, the exporter refuses to start when scrape configs or rule files are present.

<ParamField path="global.external_labels.project_id" type="string" required>
  The Google Cloud project ID to export metrics to. This is the GCP project that owns the Cloud Monitoring workspace. Must be non-empty.
</ParamField>

<ParamField path="global.external_labels.location" type="string" required>
  The Google Cloud region (e.g. `us-central1`) or zone (e.g. `us-central1-a`) that identifies where this Prometheus instance is running. Used as the `location` dimension on all exported time series.
</ParamField>

<Warning>
  `location` cannot be set to `"global"`. You must choose a named Google Cloud region or zone that is physically nearest to your cluster. Setting `location: global` causes the exporter to return an error and refuse to export data.
</Warning>

<Tip>
  A third label, `cluster`, is optional but strongly recommended when you run multiple Prometheus instances or clusters in the same project and location. It disambiguates time series that would otherwise collide in Cloud Monitoring.
</Tip>

## Full example

The following is the complete example configuration shipped with the GMP Prometheus fork at `documentation/examples/prometheus.yml`:

```yaml prometheus.yml
# my global config
global:
  scrape_interval: 15s # Set the scrape interval to every 15 seconds. Default is every 1 minute.
  evaluation_interval: 15s # Evaluate rules every 15 seconds. The default is every 1 minute.
  # scrape_timeout is set to the global default (10s).
  # GMP requires settings project_id and location labels.
  external_labels:
    project_id: example-project
    location: us-central-1

# Alertmanager configuration
alerting:
  alertmanagers:
    - static_configs:
        - targets:
          # - alertmanager:9093

# Load rules once and periodically evaluate them according to the global 'evaluation_interval'.
rule_files:
  # - "first_rules.yml"
  # - "second_rules.yml"

# A scrape configuration containing exactly one endpoint to scrape:
# Here it's Prometheus itself.
scrape_configs:
  # The job name is added as a label `job=<job_name>` to any timeseries scraped from this config.
  - job_name: "prometheus"

    # metrics_path defaults to '/metrics'
    # scheme defaults to 'http'.

    static_configs:
      - targets: ["localhost:9090"]
```

## Configuration sections

### global

The `global` block sets default values that apply to all other configuration contexts. The two most important fields for GMP are `scrape_interval`, `evaluation_interval`, and the `external_labels` map.

- **`scrape_interval`** — How often Prometheus scrapes targets. The default is `1m`; the example uses `15s`.
- **`evaluation_interval`** — How often recording and alerting rules are evaluated. Defaults to `1m`.
- **`external_labels`** — Labels attached to all time series scraped or generated by this instance before they are exported. GMP reads `project_id`, `location`, and `cluster` from this map to populate the monitored resource on every exported time series. Values set here take precedence over the equivalent CLI flags (`--export.label.project-id`, `--export.label.location`, `--export.label.cluster`).

### scrape_configs

The `scrape_configs` block defines what targets Prometheus scrapes and how. Each entry has at minimum a `job_name`. GMP exports every scraped time series to Cloud Monitoring unless you apply metric filtering via the `google_cloud.export.match` setting.

### alerting

The `alerting` block configures one or more Alertmanager endpoints. Prometheus pushes firing alerts to these endpoints. This section is unchanged from upstream Prometheus and has no GMP-specific requirements.

### rule_files

The `rule_files` block lists paths (or glob patterns) to YAML files containing recording and alerting rules. Prometheus evaluates these at `evaluation_interval`. When at least one rule file is configured, the GMP exporter validates that `project_id` and `location` are set — it returns an error at startup if they are missing.

## Hot-reload and the google_cloud section

Prometheus reloads its configuration when it receives a `SIGHUP` signal (or when you send an HTTP `POST /-/reload` request). The GMP-specific settings nested under `google_cloud` in `prometheus.yml` support hot-reloading, which means you can change export credentials, compression, or metric filtering without restarting the process.

```yaml prometheus.yml (with google_cloud section)
global:
  external_labels:
    project_id: my-project
    location: us-central1
    cluster: prod-cluster

google_cloud:
  export:
    compression: gzip
    credentials: /etc/secrets/sa-key.json
```

On each reload, the exporter applies the new `google_cloud.export` values and, if the credentials file or compression setting changed, recreates the internal gRPC metric client. This avoids a full process restart in security-sensitive Kubernetes environments where credentials are rotated via mounted secrets.

<Note>
  The `google_cloud` section only controls a subset of exporter settings — those that are safe to change at runtime. Options such as `--export.disable` and `--export.debug.disable-auth` can only be set at startup via CLI flags and are never overridden by a hot-reload.
</Note>
